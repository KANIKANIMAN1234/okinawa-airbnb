# システム仕様書：家族収支管理アプリ (v6.5.0)

## 1. システム概要
本システムは、LINE LIFFプラットフォーム上で動作する家族用収支管理ソリューションである。ユーザーはLINEアプリから直接、日々の入金および出金を記録し、家族間でデータを共有できる。画像圧縮技術と外部ホスティング（Vercel）の活用により、Googleプラットフォームの制限を回避しつつ、高速かつリッチなユーザー体験を提供する。

---

## 2. システムアーキテクチャ
本システムは、3層構造のサーバーレスアーキテクチャを採用している。



### 2.1 フロントエンド (Presentation Layer)
- **ホスティング:** Vercel
- **技術スタック:** HTML5, CSS3, JavaScript (ES6+), Chart.js
- **役割:** UIのレンダリング、LINEユーザー認証、画像のリサイズ・圧縮、データのフィルタリング、グラフ描画。

### 2.2 バックエンド (Application Layer)
- **プラットフォーム:** Google Apps Script (GAS)
- **役割:** APIエンドポイントの提供、Googleドライブへの画像保存、スプレッドシートへのデータ書き込みおよび読み取り。

### 2.3 データ・ストレージ (Data Layer)
- **データベース:** Google Sheets (スプレッドシート)
- **画像ストレージ:** Google Drive (指定フォルダ)
- **認証基盤:** LINE Developers (LIFF)

## 3. 機能詳細仕様

### 3.1 認証・初期化処理
1. **LIFF初期化:** アプリ起動時に `liff.init()` を実行。
2. **ログインチェック:** `liff.isLoggedIn()` で認証状態を確認。未ログイン時は `liff.login()` を実行。
3. **プロファイル取得:** `liff.getProfile()` を実行し、`userId` および `displayName` を取得。
4. **自動ユーザー登録:** 初回起動時、バックエンド経由で `users` シートに情報を照会。未登録の場合は自動登録。

### 3.2 収支入力機能
- **収支タイプ切り替え:** 「出金」「入金」をトグル選択。
- **カテゴリ動的制御:** 収支タイプに連動してプルダウン内容を変更。
    - **出金カテゴリ:** 電気代, 水道代, ガス代, CATV代, 外食, 通信費, ガソリン代, レンタカー代, 高速代, 交通費, 冠婚葬祭, 実家修繕, 旅費交通費, 宿泊代, 飲食代, 他
    - **入金カテゴリ:** 給与, 賞与, 副業, 還付金, お小遣い, 年金, 冠婚葬祭, 他
- **金額入力:** `inputmode="numeric"` による数値入力。
- **支払先/内容:** 任意文字列入力。
- **日付選択:** 標準カレンダーUI。初期値は `YYYY-MM-DD`（当日）。

### 3.3 画像処理・アップロード仕様
本システムの中核的な技術仕様である。

- **複数枚対応:** 1レコードに対し最大3枚まで登録可能。
- **画像ソース:** 「カメラ起動（`capture="environment"`）」または「ファイル/ギャラリー選択」から取得。
- **リサイズ・圧縮ロジック:**
    - `FileReader` で画像を `DataURL` として読み込み。
    - `Canvas API` を使用し、横幅最大1024pxに縮小（アスペクト比維持）。
    - `toDataURL('image/jpeg', 0.7)` により画質を70%に圧縮。
- **プレビュー:** アップロード前にインライン表示。個別の削除ボタンで配列から削除可能。

### 3.4 履歴管理・フィルタリング仕様
- **データ取得:** バックエンドから `getList` アクションで全件取得。
- **フィルタリングロジック:** フロントエンド（JavaScript）側で以下の4条件の積（AND検索）を実行。
    - `User`: 登録名による一致。
    - `Category`: カテゴリ名による一致。
    - `Year`: 日付オブジェクトの `getFullYear()` による一致。
    - `Month`: 日付オブジェクトの `getMonth() + 1` による一致。
- **収支合計計算:** - `Type == "入金"` の場合は加算、`Type == "出金"` の場合は減算して算出。

### 3.5 分析・統計仕様 (サマリー)
- **Doughnut Chart (支出内訳):**
    - 支出データのみを抽出。
    - カテゴリごとに合計額を算出し、構成比をグラフ化。
    - 中央テキストプラグイン: `afterDatasetsDraw` フックを使用し、ドーナツ中央に「ユーザー名」および「対象期間」を動的に描画。
- **Vertical Bar Chart (収支バランス):**
    - 「入金合計」と「出金合計」を2本の棒グラフで比較表示。
- **判例表示:** 各カテゴリの色と名称をリスト表示。

## 4. データ構造仕様 (Google Sheets)

| 列 | ヘッダー名 | データ型 | 備考 |
| :--- | :--- | :--- | :--- |
| A | 登録日時 | Date | GAS側で `new Date()` を付与 |
| B | 利用日 | Date String | ユーザー選択の日付 (YYYY-MM-DD) |

| 列 | ヘッダー名 | データ型 | 備考 |
| :--- | :--- | :--- | :--- |

## 5. API仕様 (GAS / doPost)

### 5.1 リクエスト形式 (JSON形式)
```json
{
  "action": "アクション名",
  "data": { ...各種パラメータ... }
}```


### 5.2 アクション定義 (GAS Backend Logic)
GAS側（`doPost`関数）で受け取ったリクエストの `action` パラメータに基づき、以下の内部関数を実行する。

#### 5.2.1 register (ユーザー登録・照会)
* **概要:** アプリ起動時にユーザーの存在を確認し、新規ユーザーであればデータベースへ登録する。
* **処理フロー:**
    1. `users` シートの全データを取得。
    2. リクエストに含まれる `lineId` と既存のA列（LINE ID）を照合。
    3. 存在しない場合、A列に `userId`、B列に `displayName`、D列に実行時の `new Date()` を追記。
* **返却値:** `"OK"` またはステータスメッセージ。

#### 5.2.2 upload (収支データおよび複数画像の保存)
* **概要:** 入力された収支情報と、最大3枚の画像データを一括で処理し、記録する。
* **画像処理詳細:**
    1. リクエスト内の `imageBlobs` 配列（Base64形式）をループ処理。
    2. `Utilities.base64Decode` を用いてバイナリデータに変換。
    3. 各画像に対して `Utilities.newBlob` を生成（MIMEタイプは `image/jpeg` 固定）。
    4. Googleドライブの `FOLDER_ID` で指定されたフォルダ内に `createFile` で保存。
    5. 保存したファイルの `setSharing(DriveApp.Access.ANYONE_WITH_LINK, DriveApp.Permission.VIEW)` を実行し、閲覧権限を付与。
    6. 生成された公開URLを一時配列に格納。
* **データベース書き込み:**
    1. 取得したURL配列をカンマ（`,`）で結合し、1つの文字列（`fileUrls`）にする。
    2. `data` シートの末尾行に以下の順序でデータを追加（`appendRow`）。
        * [A] タイムスタンプ, [B] 利用日, [C] 利用者名, [D] 金額, [E] カテゴリ, [F] 支払先/内容, [G] 結合済み画像URL, [H] LINE ID, [I] 収支区分（入金/出金）
* **返却値:** `"Success"`。

#### 5.2.3 getList (全履歴データの取得)
* **概要:** 履歴表示およびサマリー集計のために、スプレッドシートから全データを取得する。
* **処理フロー:**
    1. `data` シートの有効範囲（`getDataRange`）から全値を取得。
    2. 1行目（ヘッダー）を除外。
    3. `reverse()` を実行し、新しいデータが先頭に来るように並べ替え。
    4. 各行をオブジェクト形式にマッピング（日付型は `Utilities.formatDate` で `yyyy-MM-dd` 文字列に変換）。
    5. 画像URL列（G列）については、再度カンマで分割して配列に戻す。
* **返却値:** フォーマット済みオブジェクトの配列。

#### 5.2.4 getUsers (フィルタ用ユーザー名抽出)
* **概要:** 画面上のフィルタプルダウンを動的に生成するためのユーザー一覧を取得。
* **処理フロー:**
    1. `users` シートからB列（名前）を取得。
    2. 重複を排除した名前のリストを生成。
* **返却値:** ユーザー名の文字列配列。

## 6. 技術的制約・セキュリティ仕様

### 6.1 通信・認可
* **CORS (Cross-Origin Resource Sharing):** VercelドメインからGoogleドメインへのクロスドメイン通信を許可するため、GASは `ContentService` を使用して適切なMIMEタイプ（JSON）でレスポンスを返す。
* **公開設定:** GASのウェブアプリは「実行ユーザー：自分」「アクセスできるユーザー：全員（Anyone）」としてデプロイ。

### 6.2 ペイロード・容量制限
* **画像サイズ制限:** GASの `POST` リクエストには容量制限（約10MB〜50MB）がある。高画質な写真を3枚そのまま送るとエラーになるため、フロントエンド側で以下の制約を設ける。
    * 最大横幅: 1024px（Canvasリサイズ）
    * 圧縮率: 0.7 (JPEG 70%)
* **実行時間制限:** GASの1リクエストあたりの実行時間は最大6分。本システムのアップロード処理は通常数秒で完了する設計となっている。

### 6.3 エラーハンドリング
* **フロントエンド:** `fetch` 通信が失敗した際、またはGASから `status: "error"` が返った際、画面下部のログエリア（`log-box`）にエラー詳細を表示し、送信ボタンを再活性化する。
* **バックエンド:** `try-catch` ブロックを全アクションに適用。予期せぬエラー（スプレッドシートIDの間違い等）が発生した際は、スタックトレースをJSON形式で返却し、原因特定を容易にする。

## 7. 運用・メンテナンス仕様

### 7.1 ストレージ管理
* **Googleドライブ:** 画像が保存されるフォルダの容量は、実行ユーザーのGoogleストレージ制限（無料版15GB）に依存する。定期的な不要画像の削除、またはアーカイブを推奨。
* **スプレッドシート:** スプレッドシートの最大セル数（現在1000万セル）に達しないよう、1年ごとにデータの別シート退避（アーカイブ）を行う運用を想定。

### 7.2 バージョン管理
* **GASの更新:** `Code.gs` のコードを修正した際は、必ず「新しいデプロイ」または「デプロイの管理」から**新しいバージョン**を発行しなければ反映されない。
* **フロントエンドの更新:** GitHubへのPushによりVercelが自動ビルドを実行。デプロイ完了後、LINE LIFF側でキャッシュが残っている場合は、LINEアプリを一度完全に終了させて再起動する必要がある。
