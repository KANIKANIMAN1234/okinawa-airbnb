## 1. プログラム構成
本システムは、クライアントサイド（Vercel）で動作するフロントエンド・ロジックと、サーバーサイド（GAS）で動作するバックエンド・ロジックの2つの主要な命令群で構成される。

---

## 2. フロントエンド詳細設計 (JavaScript)

### 2.1 グローバル変数定義
| 変数名 | 型 | 用途 |
| :--- | :--- | :--- |
| `LIFF_ID` | String | LINE Developersで発行されたLIFF識別子。 |
| `GAS_URL` | String | デプロイ済みのGASウェブアプリURL（/exec）。 |
| `currentType` | String | 現在選択中の収支タイプ（"出金" または "入金"）。 |
| `imageBlobs` | Array | 選択された圧縮済み画像（Base64）を保持する配列（最大3要素）。 |
| `allData` | Array | GASから取得した全件レコードのキャッシュ。 |
| `myChart` | Object | Chart.jsのドーナツグラフインスタンス。 |
| `myBarChart` | Object | Chart.jsの垂直棒グラフインスタンス。 |

### 2.2 主要関数ロジック

#### 2.2.1 画像圧縮・リサイズ処理 (`handleFiles`)
ユーザーがカメラまたはギャラリーから写真を選択した際にトリガーされる。
1. **入力チェック:** 既に `imageBlobs.length` が3以上の場合は処理を中断。
2. **非同期読込:** `FileReader.readAsDataURL` を用い、ファイルをBase64文字列として取得。
3. **Canvasリサイズアルゴリズム:**
   * `Image`オブジェクトの `onload` イベント内で、元画像の `width` / `height` を取得。
   * `MAX_WIDTH = 1024` を基準とし、アスペクト比を維持したまま `height` を算出。
   * `document.createElement('canvas')` でメモリ上にキャンバスを作成し、算出サイズをセット。
   * `drawImage()` で縮小描画。
4. **出力:** `canvas.toDataURL('image/jpeg', 0.7)` を実行。画質を70%に落とすことで、1枚あたりの容量を数百KB程度に抑制し、GASの通信制限を回避する。



#### 2.2.2 データ送信処理 (`sendData`)
1. **バリデーション:** 金額（`amount`）が未入力の場合、`alert` を表示し処理を中断。
2. **ペイロード構築:** 以下のオブジェクトを生成。
   * `lineId`, `userName`, `type`, `amount`, `category`, `shop`, `date`, `imageBlobs`
3. **非同期通信:** `fetch(GAS_URL)` に対し、`method: "POST"`, `mode: "no-cors"` は使用せず標準のPOSTリクエストを送信。
4. **完了処理:** `res.status === "success"` 受信後、`location.reload()` を実行してステートを完全に初期化。

#### 2.2.3 グラフ描画プラグイン (`centerTextPlugin`)
Chart.jsの `doughnut` グラフの中央に文字を描画するためのカスタムロジック。
* **描画タイミング:** `afterDatasetsDraw` フックを使用。
* **座標算出:** `chart.getDatasetMeta(0).data[0]` の中心座標（x, y）を取得。
* **描画内容:**
  * 上段（Bold 14px）: 選択中のユーザー名 ＋ [カテゴリ名]
  * 下段（11px）: 選択中の期間（YYYY年MM月）

## 3. バックエンド詳細設計 (Google Apps Script)

### 3.1 エントリポイント (`doPost`)
1. **リクエスト解析:** `JSON.parse(e.postData.contents)` により、フロントからのJSONをオブジェクト化。
2. **アクションルーティング:** `params.action` に基づき、各内部関数へ処理を委譲。
3. **レスポンス生成:** `ContentService.createTextOutput` を用い、`MimeType.JSON` 形式でシリアライズ。

### 3.2 データ永続化ロジック (`uploadData`)
1. **Googleドライブ保存:**
   * `payload.imageBlobs` をループ。
   * `Utilities.base64Decode` によりバイナリ（Blob）へ変換。
   * `DriveApp.getFolderById(FOLDER_ID).createFile(blob)` で保存。
   * 生成されたURLを配列にプッシュ。
2. **スプレッドシート書き込み:**
   * URL配列を `join(",")` で文字列化。
   * `SpreadsheetApp.openById(SPREADSHEET_ID).getSheetByName('data').appendRow([...])` により、[登録日時, 利用日, 名前, 金額, 分類, 店名, URL, ID, タイプ] の順で最終行に追記。



### 3.3 履歴取得ロジック (`getListData`)
1. **データ取得:** `getDataRange().getValues()` で全行（二次元配列）を一括取得。
2. **加工処理 (`map`関数内):**
   * 日付データ: `Utilities.formatDate` により、クライアントが扱いやすい `yyyy-MM-dd` 形式に強制変換。
   * 画像URL: 文字列を `.split(",")` し、再び配列形式に戻すことで、フロント側の `map` 処理に対応させる。
3. **並び替え:** `reverse()` を適用し、降順（最新の投稿が上）として返却。

## 4. 異常系・例外処理仕様

### 4.1 通信エラーハンドリング
* **タイムアウト対策:** GASの実行制限（6分）に対し、フロント側で画像圧縮を行うことで、通信ペイロードを極小化。
* **Fetch失敗時:** `catch(e)` 内で `addLog` 関数を呼び出し、ユーザーがLINE画面上でエラー内容を確認できるように実装。

### 4.2 データベース整合性
* **同時書き込み:** GASの `appendRow` はアトミックに動作するため、家族メンバーが同時に送信しても行の競合（上書き）は発生しない。
* **データ型保証:** スプレッドシート側で金額列を「数値形式」として保持できるよう、送信前に `Number(amt)` による型キャストを徹底。

## 5. UI/UX制御ロジック
* **タブ切り替え (`showPage`):** CSSの `display: none` と `display: block` を切り替え。ページ遷移なしの高速な画面切り替えを実現。
* **送信ボタン制御:** 送信開始時に `btn.disabled = true` を設定し、多重投稿および連打によるエラーを物理的に防止。
